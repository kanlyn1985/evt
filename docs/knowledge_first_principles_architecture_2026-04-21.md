# 知识库第一性原理架构设计

## 1. 文档目的

本文档不讨论某个具体问句，也不讨论某个具体 PDF。  
目标是从第一性原理回答这几个根问题：

1. 什么是知识
2. 知识库真正应该存什么
3. 文档、表格、图片、时序在知识库中分别是什么角色
4. 一个通用知识库应该如何实现准确召回
5. 当前 KB1 应该如何从现状演进到真正的知识驱动系统

这份文档是后续所有实现工作的上位约束。

---

## 2. 第一性原理：什么是知识

### 2.1 知识不是文本

文本、表格、图片、公式、图示，都只是知识的载体。  
它们本身不是知识。

知识至少满足以下条件：

- 能描述现实中的对象、状态、关系、约束或变化
- 能在不同表达方式中保持同一语义
- 能支持检索、解释、复用和推断
- 能被放在上下文中理解，而不是孤立存在

所以：

- 一段 OCR 文本不是知识
- 一张表格不是知识
- 一页 PDF 也不是知识

它们只是知识的观察结果。

### 2.2 知识库的本质

知识库的本质不是“存文档”，而是：

**从文档中恢复一个可计算、可检索、可推断的领域世界模型。**

---

## 3. 知识的五个基本组成

一个通用知识库至少要表达以下五类内容：

### 3.1 实体

回答“有什么东西存在”。

例如：

- 充电机
- 车辆插头
- 车辆插座
- 控制导引电路
- 检测点 1 / 2 / 3
- R4c'
- V2G

### 3.2 属性

回答“这些对象有什么特征”。

例如：

- R4c' 的标称值是 1000 Ω
- U2 上拉电压为 12 V
- 某接口支持模式 4

### 3.3 关系

回答“这些对象如何相互关联”。

例如：

- R4c' 属于控制导引电路
- 检测点 2 属于 CC 回路
- 某表描述某个系统
- 某标准替代某旧标准

### 3.4 规则 / 约束

回答“在什么条件下什么成立”。

例如：

- 某状态下检测点电压应处于某范围
- 某模式下必须具备锁止装置
- 故障发生后必须进入停机状态

### 3.5 过程 / 演化

回答“系统如何变化”。

例如：

- 握手阶段
- 预充阶段
- 能量传输阶段
- 故障停机
- 紧急停机

---

## 4. 文档只是知识的投影

文档里的各种结构，在知识库中不应被直接当成知识本体。

### 4.1 文本段落

文本段落是知识表达，不是知识对象本身。

### 4.2 表格

表格是属性、约束和关系的压缩视图。  
表本身重要，但更重要的是：

- 表头表达了属性维度
- 行表达了知识实例
- 单元格表达了对象、状态、值之间的对应关系

### 4.3 图片 / 图示 / 电路图

图示往往表达：

- 对象组成
- 连接关系
- 状态切换
- 回路归属

图不能只作为图片存储，而应映射为结构关系。

### 4.4 时序表 / 控制流程

时序类内容本质上是“过程知识”，不是普通表格，也不是普通段落。  
它们应该表达：

- 状态
- 触发条件
- 迁移
- 动作
- 时间约束

---

## 5. KB1 未来应采用的核心抽象

未来的 KB1 不应该围绕“定义、参数、表格、时序”这些表现形式组织。  
更合理的核心抽象是下面四层。

### 5.1 Concept

概念层，回答“这是什么”。

例如：

- 控制导引
- 车辆适配器
- V2G
- 锁止装置

### 5.2 Entity / Component

对象层，回答“系统由什么组成”。

例如：

- CC1
- CC2
- 检测点 2
- R4c'
- S2'

### 5.3 Constraint / Capability / State

规则层，回答“在什么条件下具有什么属性”。

例如：

- 某参数值
- 某状态约束
- 某能力要求
- 某兼容性要求

### 5.4 Transition / Process

过程层，回答“系统如何从一个状态变为另一个状态”。

例如：

- 握手
- 预充
- 充电启动
- 正常结束
- 紧急停机

---

## 6. 通用知识库的正确检索目标

检索目标不应该是：

`query -> 找到相似文本`

而应该是：

`query -> 找到对应的知识子图`

也就是说，用户问题本质上总在问以下几类事情之一：

- 这是什么 -> 概念
- 有哪些组成 -> 对象
- 有什么属性/约束 -> 规则
- 怎么变化 -> 过程

所以检索应该围绕以下四类空间展开：

- 概念检索
- 对象检索
- 规则检索
- 过程检索

---

## 7. 为什么现在会不准

当前系统的主要问题不是“有没有 OCR”，也不是“有没有向量检索”，而是：

### 7.1 用文档结构代理知识结构

当前大量逻辑仍然在使用：

- 页
- 块
- 表
- 标题

来近似知识结构。

但：

- 文档结构不等于知识结构
- 表格结构不等于语义结构
- 页面相邻不等于知识相邻

### 7.2 检索过度依赖文本命中

这会天然导致：

- 目录压正文
- 前言压参数
- 大章节压目标表
- 局部高频词污染结果

### 7.3 wiki 和 graph 还没有成为召回主链的一部分

当前 wiki 和 graph 存在，但更多是产物，不是检索骨架。

---

## 8. wiki 与 graph 的正确角色

### 8.1 wiki 的角色

wiki 不应只是可读页面。  
它应该是：

**概念归并层**

作用包括：

- 把同一概念的不同表达统一
- 建立稳定的主题入口
- 将文档中的分散知识汇聚到概念页

wiki 页的类型未来不应只限于：

- 标准页
- 术语页

还应包括：

- 主题页
- 回路页
- 参数簇页
- 过程页

### 8.2 graph 的角色

graph 不应只是附属边集合。  
它应该是：

**可计算的领域模型**

作用包括：

- 查询约束扩展
- 候选子图生成
- 知识一致性校验
- 跨章节/跨附录推理

未来 graph 至少应表达：

- concept -> entity
- entity -> parameter
- entity -> state
- process -> step
- parameter -> belongs_to_loop
- parameter -> belongs_to_interface
- parameter -> measured_at_detection_point
- table -> describes_entity
- section -> describes_concept
- document -> defines/references/replaces

---

## 9. 目标架构：从文档处理系统升级为知识系统

未来推荐架构：

```text
文档观察层
  -> OCR / layout / table / figure / reading order

语义解释层
  -> concept / entity / constraint / process / relation

知识图层
  -> wiki + graph + typed facts

查询理解层
  -> 问题类型识别 + 约束提取 + 子图定位

多通道检索层
  -> concept / entity / constraint / process / text evidence

重排与校验层
  -> 类型一致性 + 图邻域一致性 + 约束一致性

答案组装层
  -> 概念答案 / 参数答案 / 时序答案 / 规则答案
```

---

## 10. 查询不该围绕表现形式，而该围绕知识意图

例如：

### 10.1 `CC阻值有哪些`

本质是：

- 对象：CC 回路
- 目标：属性 / 参数集合

### 10.2 `CP 时序`

本质是：

- 对象：CP / 控制导引
- 目标：过程 / 状态变化 / 时序约束

### 10.3 `什么是 V2G`

本质是：

- 对象：V2G
- 目标：概念定义

所以系统应先识别知识意图，再决定召回空间。

---

## 11. KB1 的演进原则

### 11.1 停止继续围绕单个问句打补丁

任何“查不到 -> 加一个 synonym / 加一个 if”的做法都不应继续成为主方向。

### 11.2 以领域知识模型为中心

后续所有 schema、facts、wiki、graph 和检索设计，都应围绕领域模型展开。

### 11.3 先建模，再扩实现

在继续大规模写代码之前，先明确：

- 核心概念
- 核心对象
- 状态集合
- 过程集合
- 关系类型

### 11.4 以失败样例驱动验证，但不以失败样例驱动架构

失败样例用于评估系统，而不是决定系统的顶层结构。

---

## 12. 对当前 KB1 的具体建议

### 第一优先级：知识模型建模

输出领域本体模型：

- Concept
- Entity
- Parameter
- Constraint
- State
- Process
- Relation

### 第二优先级：重构 graph schema

图谱应从当前少量 document-level 边，扩展到领域级关系图。

### 第三优先级：扩展 wiki 类型

wiki 由“标准页/术语页”扩展为“主题页/回路页/过程页/参数页”。

### 第四优先级：改查询路由

从：

`query -> facts/evidence`

升级为：

`query -> wiki concept resolution -> graph expansion -> multi-space retrieval`

### 第五优先级：改答案校验

回答前必须检查：

- 证据类型是否匹配
- 子图是否一致
- 约束是否冲突

---

## 13. 实施顺序建议

### Phase 1

产出领域知识模型文档  
不要先写更多补丁。

### Phase 2

升级 schema：

- parameter 增加归属字段
- process / timing 单独建模
- graph 扩边

### Phase 3

升级 wiki：

- topic wiki
- loop wiki
- process wiki

### Phase 4

重构 retrieval：

- concept space
- entity space
- constraint space
- process space
- evidence space

### Phase 5

重构 answer：

- 概念答案
- 参数答案
- 过程答案
- 规则答案

---

## 14. 最终结论

KB1 如果要成为“通用且准确”的知识库，必须从“知识是什么”出发，而不是从“表格怎么抽、参数怎么搜”出发。

真正的方向是：

**把文档处理系统，升级为领域世界模型恢复系统。**

也就是：

- 文档只是观察层
- 知识图才是核心层
- wiki 是可读表示
- graph 是可计算表示
- 查询不是搜文本，而是定位知识子图

这才是后续实现的正确起点。
